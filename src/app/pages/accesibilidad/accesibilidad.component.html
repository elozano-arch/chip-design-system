Accesibilidad

Guía de accesibilidad para el sistema de diseño CHIP 2.0. Todos los componentes y páginas deben cumplir con los estándares WCAG 2.1 nivel AA y con la Resolución 1519 de 2020 del Ministerio de Tecnologías de la Información y las Comunicaciones de Colombia, que establece los lineamientos y estándares de accesibilidad web para entidades del Estado colombiano.

Esta guía proporciona principios, patrones de código y una lista de verificación para que los desarrolladores construyan interfaces accesibles e inclusivas.

Contraste de color

Un contraste adecuado entre texto y fondo es esencial para que personas con baja visión o daltonismo puedan leer el contenido. Las WCAG 2.1 nivel AA exigen:

  • 4.5:1 de relación de contraste para texto normal (menos de 18px o menos de 14px en negrita).
  • 3:1 de relación de contraste para texto grande (18px o superior, o 14px en negrita o superior).
  • 3:1 de relación de contraste para componentes de interfaz y elementos gráficos.

Utilice herramientas como WebAIM Contrast Checker o la extensión de navegador axe DevTools para verificar el contraste de sus componentes.

Navegación por teclado

Todos los elementos interactivos deben ser operables mediante teclado. Los usuarios que no pueden usar un ratón dependen completamente de la navegación por teclado.

Tecla Acción
Tab Mover el foco al siguiente elemento interactivo en el orden del DOM.
Shift + Tab Mover el foco al elemento interactivo anterior.
Enter Activar botones, enlaces y enviar formularios.
Espacio Activar botones, casillas de verificación y opciones de radio.
Escape Cerrar diálogos modales, menús desplegables y tooltips.
↑ / ↓ Navegar entre opciones en menús, listas y selects.

Indicadores de foco

Nunca elimine el contorno de foco (outline: none) sin proporcionar un estilo alternativo visible. Los indicadores de foco deben tener al menos 3:1 de contraste con el fondo adyacente.

Orden de tabulación

El orden de tabulación debe seguir el flujo lógico de lectura (generalmente de izquierda a derecha, de arriba hacia abajo). Evite usar tabindex con valores positivos; use tabindex="0" para incluir un elemento en el flujo natural y tabindex="-1" para permitir el foco programático sin incluirlo en el orden de tabulación.

Lectores de pantalla

Los lectores de pantalla interpretan el contenido de la página y lo transmiten al usuario de forma auditiva. Para que esta experiencia sea adecuada, es necesario proporcionar información semántica correcta.

Atributos ARIA esenciales

  • aria-label: Proporciona una etiqueta de texto cuando no existe texto visible. Ideal para botones con solo íconos.
  • aria-labelledby: Referencia el id de otro elemento que sirve como etiqueta. Útil cuando el texto ya existe en la página.
  • aria-describedby: Referencia el id de un elemento que proporciona una descripción adicional (ej. instrucciones de campo).
  • role: Define el rol semántico de un elemento cuando la etiqueta HTML nativa no es suficiente (ej. role="alert", role="dialog", role="tablist").
  • alt: Texto alternativo para imágenes. Debe describir el contenido o la función de la imagen. Use alt="" para imágenes puramente decorativas.

Formularios accesibles

Los formularios son uno de los puntos de interacción más críticos. Un formulario mal etiquetado puede ser completamente inutilizable para usuarios de lectores de pantalla.

Principios clave

  • Cada campo debe tener un <label> asociado mediante el atributo for que coincida con el id del campo.
  • Agrupe campos relacionados con <fieldset> y proporcione un título con <legend>.
  • Marque los campos obligatorios con aria-required="true" y una indicación visual (asterisco con aria-hidden="true").
  • Los mensajes de error deben usar role="alert" para que los lectores de pantalla los anuncien automáticamente.
  • Vincule instrucciones y errores al campo con aria-describedby.
  • Use aria-invalid="true" en campos con errores de validación.

Enlace "Saltar al contenido"

El enlace "Saltar al contenido principal" permite a los usuarios de teclado y lectores de pantalla evitar la navegación repetitiva y acceder directamente al contenido relevante de la página. Es un requisito de la Resolución 1519 de 2020 y de las WCAG 2.1.

Gestión de foco en diálogos

Cuando un diálogo modal se abre, el foco debe moverse al interior del diálogo. Al cerrarse, el foco debe regresar al elemento que lo activó. El foco debe quedar atrapado dentro del diálogo mientras esté abierto.

  • Use role="dialog" y aria-modal="true".
  • Etiquete el diálogo con aria-labelledby apuntando al título.
  • Permita cerrar con Escape.
  • Atrape el foco con Tab dentro del diálogo.
  • Devuelva el foco al elemento disparador al cerrar.
